Why answer-sharing fails

Three students, same blueprint, same instant. A leaked answer key from one fits none of the others.
The trick is determinism: (blueprint, studentID) → the exact same paper, every time. Reproducible for auditing, yet unique per student. No two answer keys match — so there is nothing worth leaking.

Blueprint Studio — author a question once, generate infinite variants

This answers the #1 examiner question: “how do teachers actually make these?” Write a prompt with {slots}, set ranges, give a formula. Variants update live. (Formulas use a safe evaluator — no eval.)
In the 24-hour Delhi round, this means a whole new subject can be added live, in minutes — not a verbal claim, a real-time demonstration.

How it scales (and why it isn’t an AI wrapper)

The trust-critical path — generation, grading, psychometrics — makes zero API calls. It is a pure, stateless function.
blueprint.json + studentID | v [ seeded RNG ] --pure, deterministic, stateless--> unique paper | v [ auto-grader ] --numeric tolerance + MCQ--> score + misconception feedback | v [ CTT engine ] --difficulty + discrimination--> flag broken questions No server session state. No per-user database read to GENERATE a paper. => N stateless instances behind a load balancer = serve 500,000 candidates.
Why not an “AI wrapper”: the integrity-critical logic is verifiable math, runnable offline, auditable in 10 minutes. AI is optional tooling on top (e.g. an LLM suggesting parameter ranges in Blueprint Studio) — it never touches the path that decides a grade. That separation is a deliberate engineering and ethical choice, documented in DESIGN.md.
